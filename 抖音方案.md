# 抖音视频自动发布方案

> 整理时间：2026-06-04
> 用途：短视频分发 Agent 接入抖音平台的技术做法与落地规划
> 关联现状：本项目已跑通 **B站** 真实投稿（`backend/app/agents/bilibili_agent.py`），本文以其为参照设计抖音接入。

---

## 一、抖音官方做法（OpenAPI）

抖音开放平台提供完整的视频发布 OpenAPI，**支持直接发布到用户主页（不是草稿箱）**。

### 1.1 核心流程（三步）

**① OAuth2.0 拿凭证**
- 引导用户访问授权页获取 `code`
- 用 `code` 换取 `access_token` + `open_id`（`access_token` 短期有效，需用 `refresh_token` 刷新）

**② 上传视频**
```
POST https://open.douyin.com/api/douyin/v1/video/upload_video/?open_id={open_id}
Headers:
  access-token: {access_token}
  Content-Type: multipart/form-data
Body:
  video: {二进制视频文件}
```
大文件（>300MB）走分片三段：
```
POST /api/douyin/v1/video/init_video_part_upload/
POST /api/douyin/v1/video/upload_video_part/
POST /api/douyin/v1/video/complete_video_part_upload/
```

**③ 创建作品（发布）**
```
POST https://open.douyin.com/api/douyin/v1/video/create_video/
Headers:
  access-token: {access_token}
  Content-Type: application/json
Body:
{
  "video_id": "{上传返回的 video_id}",
  "text": "视频标题 #话题1 #话题2",
  "at_users": ["{用户 open_id}"],
  "poi_id": "{地点ID}",
  "micro_app_id": "{小程序ID}"
}
```

### 1.2 对接步骤

1. 注册开发者账号：访问 [抖音开放平台](https://developer.open-douyin.com/)，注册并实名认证
2. 创建应用：选择「移动应用」或「网站应用」类型
3. 申请权限：应用详情 → 能力管理 → 能力实验室，申请 **「代替用户发布内容到抖音」（`video.create.bind`）**
4. 获取凭证：审核通过后获取 `client_key` 和 `client_secret`
5. OAuth 授权：引导用户授权拿 `code` → 换 `access_token`
6. 调用 API：用 `access_token` + `open_id` 调用上传与发布接口

### 1.3 限制

- 视频大小 ≤ 4GB，超过 300MB 必须分片上传
- 视频时长 ≤ 15 分钟
- 推荐 16:9 / 竖版，720p 及以上
- 支持格式：mp4、webm
- 单用户日调用上限：参考开放平台配额

## 二、与现有 B站实现的关键差异

本项目 B站走的是 **Web 端协议 + Cookie（SESSDATA/bili_jct）**，"拿到 Cookie 就能发，零审核"。**抖音无法照搬**：

| 维度 | B站（现状·已跑通） | 抖音 |
|---|---|---|
| 鉴权 | Cookie，后端 `.env` 直接配 | **OAuth2.0**，需用户在线授权拿 `access_token` |
| 准入 | 无需申请，逆向 Web 协议即可 | **必须有审核通过的开发者应用 + `video.create.bind` 能力** |
| 凭证寿命 | 长期（失效再换 Cookie） | `access_token` 短期，需 `refresh_token` 刷新 |
| Demo 可跑通性 | 高 | **低**——卡在应用审核 + OAuth 授权回调 |

**结论**：抖音工程上比 B站多出两块成本——
1. **应用审核**（时间不可控）
2. **OAuth 授权回调 + token 刷新**（要前端跳转 + 后端回调端点 + 定时刷新）

## 三、在本项目的落地设计（对齐现有架构）

建议复刻 B站模块结构，新增对等三件套。

```
backend/app/agents/douyin_agent.py        ← 对标 bilibili_agent.py
  ├─ credentials_ready(s)                  # 检查 access_token 是否有效/未过期
  ├─ ensure_token(s)                       # 用 refresh_token 刷新（B站无此步）
  ├─ latest_local_video()                  # 可直接复用 .cache/jobs/*/out.mp4 逻辑
  └─ publish_stream(...)                   # yield stage/progress/done：upload → create → poll

backend/app/main.py
  ├─ GET  /api/distribute/douyin/status    # 对标 bilibili_status
  ├─ GET  /api/distribute/douyin/oauth     # 新增：返回授权 URL（B站无）
  ├─ GET  /api/distribute/douyin/callback  # 新增：code → token 回调（B站无）
  └─ POST /api/distribute/douyin           # SSE 流式发布，复用 EventSourceResponse 模式

backend/app/schemas.py
  └─ DouyinPublishRequest                  # 对标 BiliPublishRequest
                                           #   去掉 tid/copyright，新增 at_users/poi_id

backend/app/config.py（Settings 新增字段）
  douyin_client_key / douyin_client_secret
  douyin_access_token / douyin_refresh_token / douyin_open_id
  douyin_token_expires_at
  douyin_default_text
```

- **SSE 事件协议**：直接沿用 B站的 `stage / progress / done / failed`，前端 `distribute.html` 加一个抖音渠道按钮即可复用现有进度 UI。
- **安全约定**：`client_secret` / `access_token` / `refresh_token` 全部只存后端 `.env`，**绝不下发前端、不入版本库**（与 B站凭证策略一致）。

## 四、Demo 阶段的现实建议

1. **抖音真链路依赖应用审核，时间不可控，不适合作为 Demo 必跑项。**
   - **优先**：把抖音做成**模拟态**（沿用前端 mock 进度），架构位先占好；
   - **真链路**：继续以已跑通的 **B站** 作为「真实发布」演示主力。
2. 若必须演示抖音真发布：提前 1–2 周走审核；OAuth 用 PKCE 流程；`access_token` 进 `.env`，保持与 B站一致的安全约定。
3. 后续抽一个 **`distribute_adapter`** 统一层，让 B站 / 抖音 / 快手 对上层暴露同一套 `publish_stream` 接口，下层各自适配差异。
