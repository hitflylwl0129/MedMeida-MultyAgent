## 2. 决策落地核对

| 决策 | 实现位置 | 验证 |
|---|---|---|
| Q1=B（纯 LLM 无 fallback） | `parse_excel_v2_1.py` 末尾 `raise RuntimeError(...)` | 重试 3 轮失败直接抛错，`product_agent.run_product_pipeline` 捕获后置 FAILED |
| Q2=3 轮 | `Settings.product_parse_llm_max_rounds: int = 3` | 配置默认值 3；`max(1, int(...))` 钳制下界 |
| Q3=复用 llm_model | `parse_excel_v2_1._llm_generate_code` 调 `complete_chat(settings=s)` | 不另开 LLM_CODE_MODEL，统一走 `Settings.llm_model` |
| Q4=只动第②步 | 第④步 `_score_in_sandbox` / `_SCORE_RUNNER` 一行未改 | 打分逻辑保持确定性，可解释性卖点不丢 |
| Q5=不缓存 | 不存在任何 `.cache/parse_code/*` 写入 | 每次任务都跑一遍 LLM，行为可预测 |

### 5.4 v2.0 / v2.1 时间对照

| 指标 | v2.0 写死规则 | v2.1 LLM 1 轮 | v2.1 LLM 2 轮（怪表） |
|---|---|---|---|
| probe 耗时 | – | 0.08-0.44s | 0.08s |
| LLM gen 耗时 | – | ~3.5s | 3.55 + 7.17 = 10.7s |
| exec 耗时 | ~3-5s（沙箱内 pandas） | ~0.07-0.10s | 0.72 + 0.07 = 0.79s |

## 6. 已知限制 / 后续路线

| 限制 | 影响 | 计划 |
|---|---|---|
| LLM 服务不可用时直接 FAILED（无 fallback） | 用户上传后看到失败提示 | v2.1.1 增加紧急一键切 `hardcoded` 的 `/api/admin/parse_mode` |
| 不缓存解析代码 | 同一份表反复跑要每次 LLM | v2.2 评估是否加 `(file_md5, schema_hash)` 缓存 |
| Prompt 只 1 个 few-shot | 极少数怪表头仍要 ≥2 轮 | v2.2 收集真实 fail case 加针对性 few-shot |
| AST 围栏黑名单 | 漏 `socket._socket` 等深路径会被 import 系统调用 | 沙箱本身已隔离，黑名单仅是护栏；目前够用 |
| 第④步打分仍写死 | 行业权重需要人工调 | v3.0 留给 LLM 写打分策略（与可解释性权衡） |

## 7. 一键回退方案（生产兜底）


然后重启 backend。流水线立即回到 v2.0 写死规则路径，不需要发版。

## 9. 已知遗留 / 后续讨论

### 9.1 用例 B 暴露的"重复 SKU"问题（非 v2.1 责任，业务侧）

公网真机 job `2030785a6e87` Top 5 里出现了两个"益生菌粉"（#1 final=0.967，#3 final=0.62）。
**根因**：用户业务表里 SKU 命名不规范，存在如"益生菌粉" / "益生菌粉(罐装)" / "益生菌粉 " 等差异。
v2.1 的 LLM 出码已经做了 groupby(产品名+科室)，但**没做命名归一化**。

**建议（待你拍板是否进 v2.2）**：在 `_SYSTEM_PROMPT` 里加一条：
> `name 字段：strip 首尾空白 + 全角逗号/括号转半角 + 删除常见后缀（如"(罐)"/"(盒装)"）后做 groupby。`

成本：Prompt 多 ~30 token；收益：业务无感知地合并同源 SKU。

### 9.2 首轮 LLM gen 偏慢

公网真机 LLM gen 首轮耗时 3.18-3.55s（hy3-preview，非流式）。
**可选优化**：
- 改用 `stream_chat` 流式 → 前端可看到代码逐字符吐出（演示亮点）
- 拆 Prompt 切短 → 减 token
- 换更小模型（如 hunyuan-turbo） → 但代码生成质量会下降

当前 ~10-15s 端到端**已经在用户耐心阈值内**（沙箱时间线 UI 让等待变成"可观看的过程"），不优先动。
