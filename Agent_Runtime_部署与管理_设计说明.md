# Agent Runtime：运行在哪里、怎么部署、怎么管理

> 版本：v0.1 | 日期：2026-06-02
> 配套文档：《医药营销多Agent协同平台_技术路线方案.md》《Demo落地规划》
> 解决一个高频误区：**"解耦的 Agent" ≠ "每个 Agent 都要独立部署成一个微服务"。**

---

## 0. 一句话结论

> **Agent 的 runtime 默认就运行在「编排器所在的那个进程」里**——5 个 Agent 是同一个 Python 进程内的 LangGraph 节点（逻辑解耦的模块），共享一个 runtime，**不需要各自独立的部署空间**。
>
> 真正需要"独立部署/独立伸缩"的，**不是 Agent 本身，而是它依赖的"重/异构/强隔离"能力**（视频生成 GPU、向量库、合规规则引擎、第三方投放对接）。
>
> 推荐演进路线：**单体进程内编排（起步） → 把重活动拆成独立 Worker（长任务） → 只在确有必要时把个别 Agent 升级为独立微服务**。不要一开始就微服务化。

---

## 1. 先厘清："解耦"的两个层次

很多人看到方案里"每个 Agent 是独立服务、可独立部署"，就以为要起 5 个进程/5 个容器。其实"解耦"有两层含义，**Demo 乃至大多数生产期只需要做到第一层**：

| 层次 | 含义 | 怎么实现 | 是否需要独立部署 |
|---|---|---|---|
| **① 逻辑解耦（必须）** | 每个 Agent 有明确的**输入/输出契约**、独立的代码模块、可单测、可替换 | Python 模块 / 类 + Pydantic 契约；LangGraph 节点 | ❌ 不需要，同进程即可 |
| **② 物理解耦（按需）** | 每个 Agent 是**独立进程/服务/部署单元**，可独立伸缩、独立发布、异构语言 | 独立 FastAPI 服务 / 独立 Worker / FaaS | ✅ 需要，但只对个别 Agent |

**核心判断**：解耦带来的收益（独立迭代、可替换、可灰度）**靠"接口契约 + 模块边界"就能拿到**，不必靠"拆进程"。拆进程会引入网络、序列化、分布式事务、运维成本，是有代价的，要在确有必要时才付。

> 类比：一个团队"分工解耦"不代表每个人要单独租一间办公室。先把职责边界划清楚（逻辑解耦），共享一个办公室（同一 runtime）效率最高；只有当某个人需要无尘车间（GPU/强隔离）时，才单独给他一个空间。

## 2. "Agent Runtime" 到底指什么

一个 Agent 跑起来，runtime 要承载这几件事：

1. **编排循环**：决定下一步调哪个 Agent、是否回环、是否暂停等人审（LangGraph 状态机）。
2. **LLM 调用**：经统一网关请求大模型（这步是 IO 等待，不吃本地 CPU）。
3. **工具/能力调用**：RAG 检索、规则引擎、TTS、视频生成、投放 API（有的轻、有的重）。
4. **状态读写**：`TaskContext` 的持久化与流转（SQLite/PG）。
5. **可观测**：trace、日志、事件推送（SSE）。

**关键事实**：本场景里 Agent 的"重活"几乎都在**外部**（LLM 在模型服务、视频在 GPU 渲染、投放在第三方平台）。编排器进程本身**主要是在做 IO 等待和状态流转，非常轻**。所以把 5 个 Agent 放在一个进程里，CPU/内存压力极小——这也是"单进程起步"完全够用的根本原因。

## 3. Agent Runtime 的几种部署形态（从轻到重）

### 形态 A：单体进程内编排（**Demo / 起步强烈推荐**）

```
┌──────────────────────────────────────────────┐
│  FastAPI 进程（一个容器/一台机器）              │
│                                                │
│   Orchestrator (LangGraph)                     │
│     ├─ node:选医生   ─┐                         │
│     ├─ node:人群      │  都是同进程内的          │
│     ├─ node:话术      │  Python 函数/类          │
│     ├─ node:视频      │  (逻辑解耦)              │
│     └─ node:合规     ─┘                         │
│                                                │
└───────┬───────────────┬──────────────┬─────────┘
        │               │              │
     LLM网关         向量库(Chroma)   外部能力(Mock)
   (外部模型服务)    (嵌入式/同机)    TTS/视频/投放
```

- **runtime 在哪**：就一个进程，所有 Agent 共享。
- **部署**：1 个服务 + 1 个数据库 + （可选）1 个向量库。`docker compose up` 级别。
- **优点**：开发调试最快、无网络/序列化开销、trace 天然连续、事务简单。
- **适用**：Demo、PoC、中小规模生产（并发不高、单租户/少租户）。
- **结论**：**M1–M5 全程用这个形态即可，不要提前微服务化。**

### 形态 B：进程内编排 + 重活动拆成独立 Worker（**生产第一步演进**）

把"耗时长、吃资源、可异步"的 Agent 任务从 Web 进程剥离到后台 Worker，用任务队列解耦：

```
FastAPI(Web/编排)  ──投递任务──►  消息队列  ──►  Worker 池
   ↑ 立即返回 task_id              (Redis/         ├─ 视频生成 worker (GPU)
   │ SSE 推进度                     RabbitMQ)       ├─ 话术/合规 worker
   └────────── 状态回写 ◄──────────────────────────┘
```

- **runtime**：编排逻辑仍在 Web 进程，但**节点的执行**被派发到 Worker（Celery / **Temporal** activity）。
- **谁该拆**：④视频（GPU、分钟级）、批量话术、批量合规扫描 → 拆 Worker；①②③⑤的轻逻辑可留在编排进程。
- **优点**：Web 不被长任务阻塞；Worker 可按队列独立扩容；失败可重试/重放。
- **Temporal 特别适合**：长流程、断点续跑、人审等待数小时/数天、可重放审计——和方案里"断点续跑 + 人审关卡 + 可追溯"的诉求高度契合。

### 形态 C：个别 Agent 升级为独立微服务（**仅在确有必要时**）

只把"真的需要独立伸缩/独立发布/异构技术栈/强隔离"的那一两个 Agent 拆成独立 FastAPI 服务，编排器通过 HTTP/gRPC 调它：

```
Orchestrator ──HTTP/gRPC──► 合规审查服务 (独立部署、独立规则热更新、独立审计库)
             ──HTTP/gRPC──► 视频生成服务 (独立 GPU 集群、独立伸缩)
   其余 Agent 仍在编排进程内
```

- **何时才拆**（满足任一才考虑，不要凑数）：
  1. **资源异构**：要 GPU（视频）vs 只要 CPU（其它）——混在一起浪费 GPU。
  2. **伸缩曲线不同**：合规扫描突发量大，需单独水平扩容。
  3. **发布节奏/团队不同**：合规规则天天更新，不想每次都重启整条流水线。
  4. **强隔离/合规要求**：审计、密钥、敏感规则需独立安全域。
  5. **异构语言**：某能力只有 Java/Go SDK。
- **代价**：网络延迟、序列化、契约版本管理、分布式追踪、部署运维翻倍。**没满足上面条件就别拆。**

### 形态 D：Serverless / FaaS / 托管 Agent Runtime（可选补充）

- **FaaS**（云函数）：适合**突发、无状态、短时**的单个能力（如一次合规正则扫描）。长流程/有状态的编排不适合放 FaaS。
- **托管 Agent Runtime**（各云厂商的 Agent 平台/Bedrock AgentCore 类）：托管伸缩与会话状态，省运维，但**绑定厂商**、定制编排（复杂回环/人审）灵活性不如自建 LangGraph。医疗合规要私有化时慎选。

## 4. 推荐演进路线（与排期对齐）

| 阶段 | 形态 | 说明 |
|---|---|---|
| **Demo（M1–M5）** | **形态 A** 单体进程内 | 5 个 Agent = LangGraph 节点，一个进程，一个 compose。**当前就该这么做。** |
| **生产初期** | **形态 A + B** | 把④视频、批量任务拆到 Temporal/Celery Worker；编排和轻 Agent 仍同进程。 |
| **规模化后** | **形态 B + C（局部）** | 仅把 ⑤合规、④视频 升级为独立服务（GPU/伸缩/热更新/审计驱动）。其余继续同进程。 |

> 原则：**先合后分、按需拆分（Decompose on demand）**。过早微服务化是本类项目最常见的过度设计。

## 5. 怎么"管理"这些 Agent（不论部署形态都要做）

管理的本质不是"管进程"，而是**管契约、管版本、管可观测、管合规**。即使 5 个 Agent 同进程，下面几条也都要落地：

### 5.1 统一 Agent 契约（最重要）
- 每个 Agent 实现统一抽象接口：`run(input: TaskContext) -> AgentResult`，输入输出都是 **Pydantic Schema**（强类型、可校验）。
- 这样无论它是"同进程函数"还是"远程服务"，**编排器调用方式不变**——未来从形态 A 升到 C，编排代码几乎不改。

```python
class Agent(Protocol):
    name: str
    version: str
    def run(self, ctx: TaskContext) -> AgentResult: ...

# 同进程实现：直接调函数
# 远程实现：同一接口，内部发 HTTP——对编排器透明
```

### 5.2 注册与发现
- 维护一个 **Agent Registry**（配置/字典即可）：`name → 实现类/服务地址 + version + 开关`。
- 编排器从 Registry 取实现，**支持本地实现/远程实现热切换**（这就是"适配器接口"思想，方案 §3.2 已有 `adapters/`）。

### 5.3 版本与灰度
- 每个 Agent 带 `version`，产出写进 `TaskContext`，**决策可追溯**。
- 支持 A/B 或灰度：同一 Agent 跑新旧两版对比（尤其话术/合规规则升级时）。

### 5.4 配置与密钥
- 集中配置（模型选型、温度、规则版本、平台 Key），**与代码分离**（env / 配置中心）。
- 合规规则、违禁词库做成**可热更新**（方案明确要求），独立于代码发布。

### 5.5 可观测（Agent 全链路追踪）
- 每个 Agent 执行前后发事件（SSE 给前端 + 落 trace）。
- 用 **LangSmith / OpenTelemetry** 串起：哪个 task、哪个节点、输入输出、token、耗时、重试、审核结论。
- 这是 Demo 的演示亮点，也是生产排障与**监管举证**的基础。

### 5.6 生命周期与编排
- 启停、重试、超时、断点续跑、单节点重跑、人审 interrupt/resume——**全部交给编排层（LangGraph/Temporal）统一管**，Agent 自己不管这些。

### 5.7 资源隔离与合规
- 同进程时靠**超时 + 并发限流 + 异常隔离**（一个 Agent 失败不拖垮流程）。
- 涉及敏感数据/审计的（合规、医生隐私）按需放独立安全域（形态 C 的理由之一）。

## 6. 常见误区对照

| 误区 | 纠正 |
|---|---|
| "解耦就得每个 Agent 一个微服务" | 解耦先靠**接口契约**，同进程即可解耦；拆服务是后置的、按需的。 |
| "Agent 多 = 进程多 = 机器多" | 5 个 Agent 一个进程完全够；瓶颈在外部能力（LLM/GPU），不在编排进程。 |
| "先把架构做成微服务，免得以后改" | 过早微服务化引入分布式复杂度，拖慢 Demo。用**统一契约**保证以后能平滑拆。 |
| "Agent runtime 要专门的部署平台" | Demo 一个 FastAPI + compose 就是 runtime；托管平台是可选项，且有厂商绑定。 |
| "每个 Agent 各自管重试/状态" | 重试/状态/人审统一由编排层管，Agent 保持无状态、纯函数式更好测。 |

## 7. 落到本工程的具体建议（可直接执行）

1. **Demo 阶段**：就用**形态 A**——`backend/` 一个 FastAPI 进程，5 个 Agent 是 `agents/` 下的模块，编排在 `orchestrator/graph.py`。一个 `docker-compose`（app + sqlite/pg + 可选 chroma）。**不拆服务。**
2. **统一抽象**：现在就把 `Agent` 接口和 `adapters/` 适配器层定义好（你方案里已有雏形），保证"同进程实现"和"未来远程实现"对编排器透明。
3. **唯一值得提前考虑拆的**：④**视频生成**——因为它吃 GPU、耗时分钟级。即便 Demo，也建议把它做成"适配器 + 异步任务"形态（Mock 实现立即返回，真实实现走 Worker），这样生产期切 GPU Worker 不动主流程。
4. **生产化触发点**：当出现"合规规则要天天热更新且不想停流水线""视频并发要单独扩容"时，再分别把 ⑤、④ 升级为独立服务（形态 C），其余保持同进程。
5. **长流程可靠性**：若人审等待可能跨小时/跨天，用 **Temporal** 替代裸 LangGraph 后台任务做编排，天然支持持久化、重放、断点续跑。

## 8. 一页速查

- **Runtime 在哪**：默认在**编排器进程内**，5 个 Agent 共享。
- **要不要独立部署**：Agent 本身**不要**；只有**重/异构/强隔离的能力**（视频 GPU、合规热更新、向量库）按需独立。
- **怎么管**：统一**契约 + 注册表 + 版本/灰度 + 配置热更 + 全链路 trace + 编排层统一生命周期**。
- **最佳实践**：**单体起步、按需拆分、契约先行、可观测兜底**。
