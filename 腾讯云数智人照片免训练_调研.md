## 0.5 PoC 实测结果（v0.2 新增章节）

### 0.5.1 鉴权 + 字段完整对齐（踩坑路径）

| 一开始我以为 | 真相 |
|---|---|
| AppKey 放 HTTP Header `X-AppKey` | ❌ 错。**没有任何自定义 Header**，所有鉴权全在 URL Query |
| 单 AppKey 就够 | ❌ 错。还需要 **AccessToken**（HMAC-SHA256 的密钥，等同 SecretKey） |
| body 平铺 `{PhotoUrl, Text, ...}` | ❌ 错。必须 `{Header:{}, Payload:{...}}` 两层 |
| 字段叫 `PhotoUrl/Text/OriginalVoice` | ❌ 错。实际是 `RefPhotoUrl/InputSsml/InputAudioUrl`，`DriverType` 必填 |
| `Resolution: "720P"` | ❌ 错。是 **int 0/1**（0=720P, 1=1080P）|
| 100008 = 任务被拒（提交失败）| ❌ 错。**任务已入队成功**，只是后续 submit 被拒（典型陷阱）|

**这些坑全部固化进 `backend/app/tencent_avatar.py`**——未来上线时不会再踩。

### 0.5.2 三段实测视频（基线话术 17 秒）

基线：v1.1 Kling 路 A 跑过的失眠科普话术（122 字），照片用 `/api/doctors/{key}/image` 直接喂。

| 组 | 形象 | 驱动 | 档位 | TaskId | 实际输出 | 时长 |
|---|---|---|---|---|---|---|
| A | middle_male | Text + `male_1` | 720P | `1e71...91d9` | **800 × 1184** | 22.3s |
| B | middle_male | Text + `male_1` | 1080P | `31f4...5b8c` | **1184 × 1760** | 22.3s |
| C | senior_female | Text + `female_1` | 720P | `8953...6dee` | **800 × 1184** | 23.6s |

**关键事实**：
1. ✅ **竖屏支持**——3 个都是竖向输出（不会被强行裁成 16:9）
2. ⚠️ **不是标准 9:16**，比例约 **5:7.4 (0.676)**，介于 2:3 和 9:16 之间；推测**输出比例由照片决定**，接口不强裁
   - 抖音/视频号 9:16 = 0.5625 → **需 FFmpeg 后处理**（上下补黑边 或 中心轻微裁切）
3. ⚠️ **TTS 朗读比 v1.1 Kling 慢 31%**（17s → 22.3s）→ 需调 `SpeechParam.Speed=1.2` 抵消
4. ✅ **720P/1080P 真实分辨率不是字面值**——720P=800×1184，1080P=1184×1760
5. ✅ **1080P 文件 23 MB vs 720P 12 MB**——确认渲染时间也接近 2 倍，与文档一致

文档 FAQ #14 原文："视频制作并发：默认 5 个，所有客户共享"。

## 1. 接口能力盘点（原文要点）

### 1.1 接口本体
- **路径**：`POST https://gw.tvs.qq.com/v2/ivh/videomaker/broadcastservice/phototovideonotrain`
- **协议**：HTTPS + JSON，异步
- **能力**：一张照片（真人/卡通/动物）+ 文本或音频 → 输出口型同步说话视频
- **形象限制**：
  - 照片：JPG/PNG/BMP/WEBP，≤6MB，单边 192-4096px，宽高比 1:2~2:1
  - 必须面部清晰、单主体、无严重遮挡
- **驱动两种**：
  - `Text` 文本驱动：≤300 字，配 `TimbreKey` 选音色（要查"分页查询音色列表"）
  - `OriginalVoice` 音频驱动：2-60 秒，支持 wav/mp3/wma/m4a/aac/ogg，≤10MB
- **输出**：720P（默认）或 1080P，**默认存 7 天**，可指定 `VideoStorageS3Url` 推到我们 COS
- **附加 Prompt**：`VideoParam.Prompt` 可控制人物表现（默认"对着镜头讲话，偶尔手势，镜头固定"）

### 1.2 异步流程
```
POST .../phototovideonotrain
  → 返回 TaskId
POST .../getprogress  { TaskId }   ← 轮询（5-10s 间隔）
  → Status: COMMIT(排队) → MAKING(制作中) → SUCCESS/FAIL
  → SUCCESS: MediaUrl + SubtitlesUrl(SRT) + Duration(ms)
            + TextTimestampResult(逐字时间戳，单位 0.1μs)
```

**预估渲染时长**：720P 模式每 1 秒视频约 20 秒渲染（官方原文）。我们目标 17 秒话术 → 约 5.7 分钟渲染。比 Kling 路 A 当前的 8 分钟略快。

### 1.3 鉴权
- **不走腾讯云 V3 签名**（不是我们熟悉的 SecretId/SecretKey）
- 走的是 aPaas 平台自有的 **AppKey 体系**
- 控制台试用入口：`https://xiaowei.cloud.tencent.com/ivh#/asserts_management`（领用 15 天试用）
- **形象包受限**：通过腾讯云计费采购后，AppKey 只能在选购的形象包内切换

## 2. 与 v1.1 Kling 路 A 的对比

### 2.1 能力维度（同类竞品）

| 维度 | v1.1 Kling 路 A（当前） | 腾讯云照片免训练 |
|---|---|---|
| **核心模式** | image2video（造基础视频）+ identify-face + advanced-lip-sync（两步式） | 一步式：照片+文本/音频 → 视频 |
| **嘴形精度** | ✅ 实测好（advanced-lip-sync 精准版） | 官方宣称"口型同步"，无对比数据 |
| **照片支持** | 单张 PNG（我们用 6 张医生形象库） | JPG/PNG/BMP/WEBP，更宽 |
| **驱动方式** | 音频驱动（我们自己 TTS） | **文本或音频均可** ⭐ 数智人优势 |
| **音色** | 我们自己 TTS（腾讯云 TextToVoice） | 接口自带音色选择 |
| **字幕** | 我们自己生成 ASS 烧录 | **直接返回 SRT + 逐字时间戳** ⭐ 数智人优势 |
| **时长上限** | 无显式上限（实测 17 秒 OK） | **≤60 秒**（音频或文本转音频后） |
| **分辨率** | 1080×1920（9:16，烧字幕固定） | 720P / 1080P（比例由照片决定，**未明示是否支持 9:16 竖屏**）⚠️ |
| **渲染速度** | 实测 8 分钟 / 17 秒成片 | 推算 ≈5.7 分钟 / 17 秒成片（略快） |
| **回调** | 不支持（我们自己轮询） | **支持 CallbackUrl**（一次性，失败不重试） |
| **视频存储** | 自家 CDN | 默认 7 天，可指定 COS |

### 2.2 计费维度（最关键的差异）

| 项 | v1.1 Kling 路 A | 腾讯云照片免训练 |
|---|---|---|
| **计费单位** | 试用包剩余约 87/100 单位（按调用计） | **小时包**（按视频时长扣减） |
| **照片单价** | 当前免费试用 | **2,500 元 / 1 小时**（≈ 41.7 元/分钟） |
| **视频素材单价** | 不适用 | 5,000 元 / 10 小时（≈ 8.3 元/分钟，**6 倍便宜**） |
| **是否预付** | 当前试用模式 | **必须预付**（无按次/按秒后付） |
| **试用** | 当前 87/100 单位 | 15 天试用权限（领用） |

**算个真账（按我们当前 17 秒视频）**：
- Kling 路 A：当前免费试用，商业化后未公开报价
- 腾讯云照片免训练：17 秒 ≈ 0.28 分钟 × 41.7 = **11.8 元/次**
- 假设 demo 期跑 100 次 → 腾讯云 ≈ **1180 元**
- 假设上线后 100 个客户每天 5 次 = 500 次/天 × 30 天 × 11.8 = **17.7 万/月**

**结论**：照片素材的腾讯云方案**单价比视频素材方案贵 5 倍**——如果未来真要规模化，应该评估"视频素材定制 + 视频免训练"路径，而不是照片免训练。

### 2.3 工程对接难度

| 项 | Kling 路 A（已落地）| 腾讯云照片免训练 |
|---|---|---|
| 鉴权适配 | ✅ JWT HS256，已固化 | 需研究 aPaas AppKey 体系（与我们熟悉的 V3 签名不一样）|
| 异步轮询 | ✅ 已固化 | 同款异步，复用现有 polling 框架可，但 status 枚举不同 |
| 输出处理 | 我们自己烧字幕 | 直拿 SRT，**反而要改 composer.py 把烧字幕逻辑改成"包装 SRT"或"用其字幕"** |
| 时间戳/字幕对齐 | 我们自己做 | 接口直返逐字时间戳 ⭐ 简化 composer 代码 |
| 调试可见性 | 已对接进度（生成基础视频/识别人脸/口型同步）| Status=COMMIT/MAKING，再加 ArrayCount 排队数，可对齐到 SSE st1-st5 |

## 4. 推荐方案：备用路径 + 灰度对比

**不替换、并存**。让 v1.1 的 Kling 路 A 继续做主路径，把腾讯云照片免训练接入做**第二条 backend**：

```
backend/app/config.py
  video_backend: "kling" | "tencent_avatar" | "motion" | "aigc" | "local"
                  ↑当前      ↑新增

backend/app/orchestrator/graph.py
  if backend_mode == "kling":          return _generate_kling(state)
  if backend_mode == "tencent_avatar": return _generate_tencent_avatar(state)  ← 新增
  ...
```

### 4.1 工程落点（最小改动）

| 文件 | 用途 |
|---|---|
| `backend/app/tencent_avatar.py`（新）| JWT/AppKey 鉴权 + 提交任务 + 轮询 + 拉成片/SRT |
| `backend/app/orchestrator/graph.py` | 加 `_generate_tencent_avatar` 节点，dispatch 增加分支 |
| `backend/app/composer.py` | 用接口返回的 SRT 直接烧字幕（省一段 TTS+ASS） |
| `backend/.env.example` | 加 `TVS_APPKEY`、`TVS_TIMBRE_KEY`、`TVS_RESOLUTION` 等 |
| `backend/app/config.py` | 加 `tencent_avatar_*` 配置字段 + `tencent_avatar_ready` 属性 |
| `backend/app/main.py` | `/api/health` 增 `tencent_avatar_ready` 字段（沿用 v1.1 习惯）|

### 4.2 演进路线

| 阶段 | 动作 |
|---|---|
| **Phase 1（建议本次做）**| 申请 15 天试用 → 跑 3-5 条真实对比片 → 嘴形/字幕/合规口径打分 |
| **Phase 2**（看 P1 效果）| 拍板：① 替换为腾讯云  ② 继续 Kling 主路径  ③ 双路径并存让用户/产品选 |
| **Phase 3**（规模化）| 若选腾讯云，评估"视频素材+定制形象+小时包"路径（5× 便宜） |

### 4.3 为什么不立即替换
1. **嘴形质量对比缺失**：Kling 路 A 我们实测嘴形精度已经很好，不应在没有 A/B 对比数据时换主路径
2. **当前 Kling 试用还剩 87/100 单位**：用完再说付费切换
3. **腾讯云价格不透明优势**：2500/小时（照片）的成本，需要先评估能否压到"视频素材版"路径
4. **v1.1 已经稳定跑通**：稳定的链路不要轻易动，新接入做对照组
